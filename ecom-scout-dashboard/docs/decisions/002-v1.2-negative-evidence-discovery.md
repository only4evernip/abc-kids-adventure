# 002 - V1.2 Negative Evidence Discovery on Public Jina Paths

- **Status:** Declined (Current Path)
- **Date:** 2026-03-13

## Context

V1.1 proved that the scout pipeline could run end-to-end:

- brief parsing
- fetch cache / draft cache
- Jina-based web fetching
- Gemini/OpenAI-compatible summarizer adapters
- schema validation / downgrade guards
- final scout-card generation

The next business problem in V1.2 was to reduce overly optimistic `go-deeper` decisions by injecting **real negative evidence** from high-signal user feedback sources, especially:

- Reddit complaint threads
- Amazon critical reviews

The goal was not just technical variety. The goal was to force the model to see genuine user pain, abandonment, discomfort, and refund-risk signals.

## Actions Tried

### 1. Reddit discovery via public search

We tested a low-cost discovery path:

- primary: `s.jina.ai`
- fallback: `r.jina.ai` wrapping Google SERP pages

We also poisoned queries with explicit negative intent, for example:

- `site:reddit.com "posture corrector" reddit "don't buy"`
- `site:reddit.com "posture corrector" reddit "waste of money"`
- `site:reddit.com "posture corrector" reddit "stopped using"`
- `site:reddit.com "posture corrector" reddit pain`

### 2. Seed injection to bypass discovery instability

To separate discovery problems from evidence-quality problems, we hardcoded Reddit thread seeds and later converted them to `old.reddit.com`:

- `r/backpain/...`
- `r/stretching/...`
- `r/Posture/...`

### 3. Amazon critical reviews seed flow

We added a narrow Amazon experiment:

- derive ASIN from a known product URL
- build `filterByStar=critical` review URL
- fetch via Jina Reader
- clean and inject into summarization pipeline

## Outcomes

### What worked

- Public web article pages (for example Healthline) were fetchable via Jina Reader and usable for summarization.
- Gemini/OpenAI-compatible adapters successfully generated structured scout drafts once prompts were hardened.
- Schema guards correctly rejected malformed / invented LLM outputs.
- Public negative-evidence experiments were easy to prototype and measure.

### What failed

#### Reddit discovery

- `s.jina.ai` returned `401` for search discovery attempts.
- Google SERP fallback returned anti-bot / unusual traffic pages instead of real result pages.
- Therefore discovery produced **zero Reddit thread URLs** in the public fallback path.

#### Reddit thread body fetch

Even when discovery was bypassed through hardcoded seeds:

- `r.jina.ai` on both `www.reddit.com` and `old.reddit.com` returned access-limited shells rather than meaningful comment bodies.
- We received `200` and non-empty text, but the semantic payload was essentially “access blocked / content unavailable”, not user complaints.

#### Amazon critical reviews

- Jina Reader returned a similar blocked shell for Amazon critical review pages.
- Again, the system got `200` + text, but not the actual negative review body.

## Decision

We are **declining the current public-Jina path** for deep negative-evidence extraction from top-tier platform comment/review pages.

Specifically, we are pausing direct public-path extraction from:

- Reddit thread pages
- Amazon review pages

under the current architecture.

## Rationale

Continuing to push on this path would likely waste large amounts of time on:

- prompt tweaking
- regex tweaking
- shallow parser changes

without solving the real bottleneck, which is:

> public fetch paths cannot reliably penetrate high-value, high-protection dynamic review/comment surfaces.

This is a **source access problem**, not a schema problem and not primarily a summarizer problem.

## Consequences

### Short-term

- Default scout pipeline should return to a predictable, clean baseline.
- Experimental hardcoded seeds should not stay permanently wired into the default live path.
- We should prefer stable, fetchable sources in the next iteration.

### Medium-term options

#### Preferred near-term direction

Search for negative evidence in sources that are more likely to yield real text via Jina Reader, such as:

- independent review blogs
- niche forums
- product review roundups with explicit `cons`
- long-term review articles
- smaller discussion communities

#### Future high-effort direction

If we truly need Reddit/Amazon deep comment text, that likely requires a dedicated access layer, such as:

- browser automation
- authenticated sessions
- proxy / anti-bot infrastructure
- private scraper services

That is a separate architectural step and should not be disguised as a small V1.2 fetch tweak.

## Follow-up

- Restore default live fetch path to avoid always-on experimental seeds.
- Keep experimental adapters behind explicit flags or dev-only toggles.
- Link TODOs in fetch adapters back to this decision record so future work starts from the correct context.
