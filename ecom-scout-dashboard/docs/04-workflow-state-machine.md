# 状态机与动作流转（重构版）

## 1. 核心原则

这个工具不是复杂项目管理系统，v1 也不追求拖拽看板。

它的目标很务实：

> **让每条候选最终都落到一个明确动作。**

所以 v1 的状态机应保持轻量，优先通过：
- 表格列
- 下拉框
- 右侧详情抽屉

来完成流转，而不是一开始就上 Kanban 拖拽。

---

## 2. v1 主状态机

| 状态 | 含义 | 进入条件 |
|------|------|----------|
| 待评估 | 刚导入，系统已评分但未人工处理 | 新导入 |
| 观察池 | 值得盯，但不应立即投入 | RPS 60-77 或高风险需复核 |
| 待补证 | 信息不足，不能拍板 | 缺字段 / 数据过期 / 证据不足 |
| 供应链核价 | 已进入实操前验证 | RPS >= 78 且非高风险且非待补证 |
| 淘汰库 | 明确不做 | Gate reject / 分数过低 / 人工否决 |

### v1 暂缓状态
以下状态保留到 v2 再说：
- 打样验证
- 小单测试
- 正式立项

原因：现在先把 **发现 -> 过滤 -> 动作** 这条链打穿更重要。

---

## 3. 状态切换门槛

## 待评估 -> 观察池
满足：
- 未触发 Gate reject
- 分数 >= 60
- 但不满足“立刻推进”条件

## 待评估 -> 待补证
满足任一：
- 核心字段缺失 >= 2
- 数据过期
- 文本证据过弱

## 待评估 -> 供应链核价
必须同时满足：
- Gate pass
- RPS >= 78
- 非高风险
- 非待补证
- 存在至少 1 个明确破局点

## 任意状态 -> 淘汰库
满足任一：
- Gate reject
- RPS < 40
- 创新空间锁死
- 人工明确否决

---

## 4. v1 不强行做复杂审计，改做本地导出

本地网页没有完整账户体系，所以 v1 不强制：
- 谁改的
- 登录身份

### v1 要求改成
- 每次人工改状态时允许填写备注
- 可导出当前 Session（JSON / CSV）
- 导出时保留：
  - 状态
  - 备注
  - 标签
  - 时间

这样至少不会“刷新即失忆”。

---

## 5. 淘汰原因标签

建议固定以下标签：

- 高危禁入
- 创新空间锁死
- 过热拥挤
- 数据过期
- 待补证
- 高需低利陷阱（v2）
- 供应链无破局点（v2）
- 人工否决

---

## 6. v1 前端交互建议

### 主界面
- 一张高密度表格
- 左侧筛选器
- 右侧详情抽屉

### 状态流转方式
- 表格最后一列放一个状态 Select
- 或一个动作按钮组：
  - 移交核价
  - 标记待补证
  - 放入观察池
  - 淘汰

### 不建议 v1 先做
- 拖拽看板
- 多泳道协作视图
- 太复杂的项目管理感交互

---

## 7. 本质

这个状态机不是为了看起来像 Jira，
而是为了保证：

> **每一条候选，不是停在“分析完了”，而是进入“下一步该干嘛”。**
