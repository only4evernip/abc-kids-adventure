这份文件不是产品 Roadmap PPT，而是实际开发时的路线约束。

目标：

> **让后续开发保持收敛，始终围绕“专用工作台”初衷推进。**

# 1. 当前阶段判断

当前版本已经完成：
- 导入防覆写
- 状态解耦
- stable ID 加固
- 导入失败摘要
- 导入统计
- 自适应布局
- 导入状态机 + stepper
- 查询性能第一轮优化
- 核心测试补强
- 专用工作流高亮 + 快捷筛选
- AI 协作基础设施文档（规则 / Prompt / 项目上下文）
- Scout Card 导入 / 展示 / 单条导出
- Feishu Bitable live sync（create / update）
- Feishu 同步防呆（字段预检 / 重复 Card ID fail-fast）
- 批量导出同步 JSON
- 待处理队列筛选
- 保存并处理下一条

结论：

> **V1 闭环已完成，接下来进入“专用工作台效率优化 / 体验抛光”阶段，而不是继续补平台化大功能。**

# 2. 当前最值得做的事（按优先级）

## P1 - 队列体验抛光

### 2.1 队列剩余感知
目标：
- 让操盘手清楚知道自己当前在处理第几条、还剩多少条

建议形式：
- 抽屉里显示 `第 X / N 条`
- 表格顶部显示当前队列剩余数
- 队列清空后给更明确的完成反馈

---

### 2.2 备注结构化增强
目标：
- 让 notes 更像决策沉淀，而不是普通留言板

可选方向：
- 判断
- 风险
- 下一步动作
- 供应链结论

注意：
- 先保留纯文本存储
- 不要引入富文本
- 不要把结构做得太重

---

### 2.3 查询 profiling / 可观测性
目标：
- 知道当前筛选到底快不快
- 为未来 5 万条量级做判断依据

建议内容：
- 查询耗时
- 当前结果数
- 是否启用关键词延迟筛选

---

## P2 - 业务闭环增强

### 2.4 失败清单导出
目标：
- 把当前失败摘要升级成可导出的 JSON/CSV

### 2.5 更细的导入恢复语义
目标：
- 进一步增强导入中断后的可恢复性 / 可解释性

### 2.6 Feishu 多表联动 / 更深操盘流转
目标：
- 在单表闭环稳定运行一段时间后，再决定是否值得继续扩

---

# 3. 现在不要做的事

这些事情当前阶段应明确延后。

## 3.1 云同步
不做原因：
- 破坏本地优先优势
- 显著增加复杂度
- 当前并没有真实协作刚需验证

## 3.2 多账号 / 权限系统
不做原因：
- 偏平台化
- 与当前单用户专用定位不匹配

## 3.3 复杂图表 / BI 面板
不做原因：
- 目前核心问题不是可视化炫，而是筛选与决策沉淀

## 3.4 富文本 notes
不做原因：
- 会弄脏 JSON / Dexie 结构
- 纯文本目前足够高效

## 3.5 重型后端化 / 服务端分页架构
不做原因：
- 过早优化
- 当前体量和目标不需要

---

# 4. 进入 V1.1 的条件

以下任一情况出现时，可以启动 V1.1 扩展讨论：

1. 已经有稳定真实使用 3–7 天
2. 已经积累一批真实反馈，而不是想象需求
3. 已经明确知道“什么最影响继续处理效率”

如果没有这些前提，优先继续做小范围专用优化，不要开大版本叙事。

# 6. 每次准备开始新功能前先问的 5 个问题

1. 这个改动是不是让助理更容易继续跟进商品？
2. 这个改动会不会增加很多复杂度，但只带来一点点价值？
3. 这个改动会不会碰到人工状态 / notes / merge 保护语义？
4. 这个需求是真实使用中冒出来的，还是脑补出来的？
5. 这件事是不是应该等真实数据规模和反馈到了再做？

如果第 1 条答不上来，或者第 4/5 条明显偏虚，先别做。
